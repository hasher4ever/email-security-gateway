# Deploy

## Prerequisites

- A subdomain you control for the gateway (e.g. `mail.example.com`)
- Ability to set the corp domain's **MX** record
- A Railway project with billing (or any docker-compose host)
- API keys (free tiers): Google Safe Browsing, optional PhishTank / OpenPhish

## Phase 1 — local dry-run

Bring the stack up on a workstation to verify the config compiles before any DNS changes.

```bash
cd email_security
cp .env.example .env
$EDITOR .env                      # set CORP_DOMAIN, GATEWAY_HOSTNAME, TRUSTED_DOMAINS, GSB_API_KEY
docker compose up -d postgres redis
docker compose logs -f postgres   # wait for "database system is ready"
docker compose up -d feeds        # populates known_phish.map and lookalike_domains.map
docker compose up -d rspamd       # config-tests on build; will fail loudly if broken
docker compose up -d postfix
```

Smoke test:

```bash
# Send a test message through Postfix
docker compose exec postfix sh -c '\
    echo "Subject: hello\n\nbody" | sendmail -f from@example.org user@${CORP_DOMAIN}'
docker compose logs rspamd | grep AUDIT   # see the verdict log line
```

## Phase 1 — Railway deploy

Railway doesn't natively run docker-compose, but it accepts a Dockerfile per service.

1. Create a Railway project. Add five services:
   - `postgres` — Railway's managed Postgres plugin (cheaper than self-hosted).
   - `redis` — Railway's Redis plugin.
   - `postfix` — point at `email_security/postfix` (Dockerfile builds).
   - `rspamd` — point at `email_security/rspamd`.
   - `feeds` — point at `email_security/feeds`.
2. In each service's **Variables**, paste the contents of your `.env`. Use Railway's `${{Postgres.PGHOST}}` and `${{Redis.REDIS_HOST}}` references for the DB hosts.
3. Expose **Postfix** on TCP 25 (and optionally 587). Railway gives every service a TCP proxy port — request port 25 in the dashboard.
4. Run the SQL init manually once: `docker compose exec postgres psql -U $POSTGRES_USER -d $POSTGRES_DB -f /docker-entrypoint-initdb.d/init.sql` (or apply via Railway's psql).

## DNS cutover

When you're ready to point real mail through the gateway:

1. **Add an A record** — `mail.example.com → <Railway TCP proxy IP>`.
2. **Add reverse DNS** — Railway support ticket; SMTP rDNS is required by recipient servers to avoid being treated as spam.
3. **Generate DKIM** — `rspamadm dkim_keygen -s mail -d example.com` inside the Rspamd container. Publish the TXT record. Configure `dkim_signing.conf`.
4. **SPF** — `v=spf1 mx -all` on `example.com`.
5. **DMARC** — `v=DMARC1; p=none; rua=mailto:dmarc@example.com` while monitoring; tighten to `quarantine` then `reject` once aligned.
6. **Lower the TTL** of your existing MX record to 300s a day before cutover, then change MX to `mail.example.com`.
7. **Keep a fallback MX** with higher priority pointing at the old provider for at least a week.

## Switching from shadow to enforce

After ~2 weeks of tuning (review false positives in the BCC mailbox, add allowlist entries, watch the audit log):

```bash
# In Railway, change the Variable on Postfix and Rspamd:
GATEWAY_MODE=enforce
```

Then bounce the services. Postfix stops BCCing; Rspamd's reject/greylist actions become live.

Don't enforce until **false-positive rate is under 0.1%** in shadow mode.

## Backup

- Postgres — Railway's automated backups, daily.
- `email_security/.env` — store in your password manager; don't commit.
- `rspamd/lists/*.map` — generated by `feeds`, regenerable. Don't back up.
- Quarantine raw mails — kept in `postgres-data` (referenced from `quarantine.raw_path`). 30-day retention; security-team-only access.

## Rollback

If something goes sideways:

1. **Immediate**: change MX back to your old provider. TTL 300s = 5min.
2. **Postfix only**: stop the postfix container; senders queue and retry against the secondary MX.
3. **Rspamd only**: in `main.cf`, set `milter_default_action = accept`. Postfix delivers without filtering until you fix Rspamd.
